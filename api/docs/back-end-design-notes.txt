Here is what the rows would look like across your four tables to give them their specific powers.
1. users Table
This stores your basic identity.
id	username	email
1	Alice	alice@example.com
2	Bob	bob@example.com

2. roles Table
This defines the "buckets" of permissions.
id	name	slug
10	Administrator	admin
11	Editor	editor

3. permissions Table
These are the granular actions available in your app.
id	action_name	slug
100	Create Post	post.create
101	Delete Post	post.delete
102	Manage Users	user.manage

4. role_permissions (The Logic)
This defines what each role is allowed to do. Notice that "Admin" gets everything, while "Editor" only gets to create.
role_id	permission_id	Notes (Mental Only)
10	100	(Admin can Create)
10	101	(Admin can Delete)
10	102	(Admin can Manage Users)
11	100	(Editor can Create)

5. user_roles (The Assignment)
This defines who has which role.
user_id	role_id	Resulting Power
1	10	Alice is an Admin (Full access)
2	11	Bob is an Editor (Create only)

-----3/12/2026-----


process of how a circle is generated:
"""""""""""""""""""""""""""""""""""""

1. cicles table is created 
2. circle_details, circles_member_tracker, circles_idea_board created along side it
-> owner is added to their own circle automatically

3. conversation table is created with circle_id
3a. when an account is created, we create a already made circle, with system chats or the users first chat

4. circles_idea_board_posts is created when a user posts in the idea board
5. files_circle is inserted when a owner user updated the bg image, profile image or a user adds an image to the idea board


Notes:

conversation table:
- owner_user_id is the person who initiatiated the conversation
--> if the user conversation is 1 to 1, `to_id` is used
-->if the user conversation is a group (part of a circle), `to_id` is NOT used

- type: couple, circle
- status: active, in-active



conversation_chats:
- when the conversation is 1 to 1, end_user_id is used
- when the conversation is a circle, then messages are read by first come first serve
- conversation will have a type:
--> type: chat, announcement, system
---> chat is when a user makes a chat
---> announcement,  is when a user joines, leaves, or a change is made to the chat. When a user joins, a record will be inserted with announcement
---> system is when the system enters a record, either for tutorial or admin purposes
